# New Project Workflow

Use Orangebox to make a project controlled before making it impressive.

## Order

1. Define intent in one paragraph.
2. Create or draft an AECode Source packet.
3. Create mission boundaries:
   - allowed paths
   - forbidden paths
   - risk level
   - target output
   - proof commands
   - rollback rule
4. Choose the target language from the registry.
5. If the target is gated, ask for explicit lane authorization.
6. Create worktree/patch plan before mutation.
7. Run deterministic proof before model critique.
8. Write or cite the receipt.

## Target Selection

For backend/Ops work, use JavaScript ESM, TypeScript/Bun where already present, Python utilities, PowerShell wrappers, JSON/YAML/SQLite schemas, and Markdown docs.

For UI/web/mobile/native/deploy outputs, treat the target as future/gated unless the user explicitly opens that lane.

## Done Means

Done means:

- objective restated
- active lane named
- files touched
- commands run
- proof receipt named
- unverified assumptions called out
- rollback path named

If there is no proof receipt or command output, say it is not fully green.
