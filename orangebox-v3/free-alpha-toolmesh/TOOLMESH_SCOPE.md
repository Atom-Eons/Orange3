# ToolMesh Scope

## Purpose

Orangebox is a governed software factory. AECode is the middle language for producing outputs faster, while AtomSmasher compresses memory, work, and context. ToolMesh adds a controlled tool layer so Orangebox can use free and open tooling without losing sovereignty.

## Non-Negotiables

- TriLane stays sovereign.
- STRONGARM gates risky actions.
- Receipts prove action.
- Ghost worktrees protect the repo.
- Local-first is default.
- Cloud requires warrant.
- Tool cards are not permission to execute.
- Candidate tools do not become core until they beat baseline.
- Tool cards must declare hardware profile, execution mode, artifact policy, and template policy.
- Heavy binary outputs return file pointers and hashes only, never raw bytes.
- Workflow tools use immutable templates with variable injection, not model-generated node graphs.
- GUI-heavy tools use workspace_prep mode unless headless execution is proven.

## What ToolMesh Adds

- Tool capability cards.
- Lab-specific doctors.
- Tool routing without execution.
- VRAM and concurrency declarations.
- Artifact vault pointer policy.
- Immutable workflow template policy.
- Headless versus workspace-prep execution mode.
- Install and promotion gates.
- Artifact receipts.
- Benchmark templates.
- V3 wave preservation map.
- Alpha watchlist for new tools.

## What ToolMesh Does Not Do Yet

- It does not install ComfyUI, Goose, OpenHands, n8n, or other candidates.
- It does not call external services.
- It does not edit the visual project.
- It does not publish anything.
- It does not let tool agents touch the repo outside ghost-worktree and STRONGARM controls.
